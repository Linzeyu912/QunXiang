# 小说角色提取工具前端优化调研报告

## 一、当前项目前端现状分析

### 1.1 技术栈
| 项目 | 版本 |
|------|------|
| React | 18.2 |
| TypeScript | ~5.x |
| Vite | 5.x |
| Tailwind CSS | 3.4 |

### 1.2 现有页面与功能

当前前端仅有 **3 个页面 + 1 个组件**，采用简单的条件渲染切换（无路由）：

| 页面 | 功能 |
|------|------|
| **BookListPage（上传页）** | 上传 TXT 文件、显示书籍卡片列表、触发提取任务 |
| **HistoryPage（历史记录）** | 表格展示书籍、搜索/状态筛选、查看内容 Modal、删除 |
| **ReviewPage（审核页）** | 按状态筛选角色、Approve/Reject/Edit 操作 |

### 1.3 现存问题

1. **信息展示单一**：仅展示角色基础信息（姓名、别名、描述、置信度），缺少提取来源、章节上下文、出现频次等辅助审核信息
2. **缺少可视化**：没有提取进度可视化、角色关系图谱、章节分布统计等
3. **交互简陋**：无拖拽上传、无批量操作、无键盘快捷键
4. **无路由管理**：页面切换通过状态控制，无法分享/刷新保持状态
5. **缺少数据导出**：提取结果无法在前端导出为 JSON/CSV/Markdown
6. **响应式不完善**：表格在移动端体验差
7. **无暗黑模式**：现代工具标配功能缺失

## 二、竞品项目前端调研

### 2.1 Unstructured Platform（文档解析平台）

**定位**：企业级文档解析 ETL 平台，将非结构化文档转为结构化数据。

**前端核心功能**：
- **DAG 工作流设计器**：中央画布可视化编排处理流程（分区 → 富化 → 分块 → 嵌入 → 提取）
- **策略选择面板**：分区策略切换（Auto / Fast / High Res / VLM）
- **测试视图**：左右分栏——左侧原始文件预览，右侧 JSON 输出，支持搜索/下载
- **拖拽上传**：支持本地文件直接拖入测试
- **连接器管理**：源端（S3/GCS/Dropbox）和目标端（PostgreSQL/Pinecone）配置
- **作业调度**：定时自动运行 + 历史记录查看
- **设置面板**：结果覆盖控制、调度规则配置

**可借鉴点**：
- 提取流程的进度可视化
- 原始文本与提取结果的左右对照视图
- 拖拽上传交互
- 策略/参数配置面板

---

### 2.2 LlamaParse / LlamaCloud（文档解析 SaaS）

**定位**：GenAI 原生文档解析平台，面向 LLM/RAG 场景。

**前端核心功能**：
- **Dashboard 入口**：统一管理 6 个产品模块（Parse / Extract / Classify / Split / Sheets / Index）
- **无代码 UI（No-code UI）**：UI 优先配置，再集成 SDK
- **结构化提取界面**：基于自定义 JSON Schema 的字段提取配置
- **多格式输出预览**：Markdown / Text / JSON 切换展示
- **文件管理**：支持 130+ 种格式上传，purpose 分类（parse/extract/classify/split）

**可借鉴点**：
- 模块化 Dashboard 布局
- 多格式输出切换预览
- Schema 驱动的结构化提取配置

---

### 2.3 MinerU（开源文档解析工具）

**定位**：一站式 PDF 文档解析，支持在线 API + 离线部署 + 桌面客户端。

**前端核心功能**：
- **多终端覆盖**：Web 在线演示 + 桌面客户端 + API 文档
- **格式转换展示**：PDF → Markdown / JSON 的实时预览
- **元素级提取**：表格、公式、图片、文字分区域展示
- **API 文档集成**：完整的开发者文档 + 多语言 SDK 示例

**可借鉴点**：
- 原始文档与解析结果的同步对照
- 元素级提取结果的分区展示
- 桌面客户端作为补充（如需要）

---

### 2.4 Reducto（Agentic 文档平台）

**定位**：面向 AI 团队的文档摄取平台，强调 agentic 工作流。

**前端核心功能**：
- **结构化数据提取**：基于自定义 Schema 的字段级提取
- **文档分类**：自动检测文档类型 + 置信度展示
- **提取结果置信度可视化**：每个字段附带置信度分数
- **Schema 编辑器**：可视化定义提取字段
- **批量处理监控**：队列状态、处理进度

**可借鉴点**：
- 字段级置信度展示
- Schema 可视化编辑
- 批量任务队列监控

---

### 2.5 doccano / Label Studio（文本标注平台）

**定位**：开源数据标注工具，支持 NER、文本分类、关系抽取等任务。

**前端核心功能**：
- **项目管理界面**：创建/管理多个标注项目
- **标签编辑器**：定义标签文本、快捷键、背景色、文本色
- **序列标注视图**：文本高亮 + 标签选择 + 实体边界标注
- **关系抽取视图**：实体间连线 + 关系类型标注
- **人工审核工作流**：多轮审核、成员权限管理
- **数据导入/导出**：支持多种格式（JSONL、CoNLL、CSV）
- **进度统计**：标注完成度、成员工作量、标签分布
- **快捷键支持**：高效标注操作

**可借鉴点**：
- 文本高亮 + 实体标注视图（对审核角色来源很有用）
- 标签/状态颜色体系
- 统计仪表盘
- 批量导入导出

---

### 2.6 小说分析类项目（PlotNovel / NovelExtractor）

**定位**：面向小说文本的专项分析工具。

**前端核心功能**：
- **角色关系图谱**：节点-边图可视化角色关系
- **剧情时间线**：章节级事件脉络展示
- **世界地图归纳**：空间信息可视化
- **指定相似剧情查找**：语义搜索 + 高亮对比
- **战斗描述拆解**：特定场景的模板化提取

**可借鉴点**：
- 角色关系图谱（力导向图）
- 章节级提取来源定位
- 语义搜索与高亮

## 三、功能优化建议汇总

### 3.1 核心功能增强

| 优先级 | 功能 | 说明 | 参考竞品 |
|--------|------|------|----------|
| P0 | **提取进度可视化** | 展示 extractor → validator → entity-resolution → reviewer 各阶段进度 | Unstructured DAG |
| P0 | **原始文本对照视图** | 左右分栏：左侧章节原文，右侧提取角色，点击角色定位到原文 | Unstructured / MinerU |
| P0 | **角色来源定位** | 显示角色首次出现章节、出现频次统计、引用原文片段 | doccano 高亮 |
| P1 | **批量操作** | 批量 Approve / Reject / 删除角色 | Label Studio |
| P1 | **数据导出** | 支持 JSON / CSV / Markdown 导出提取结果 | MinerU / Label Studio |
| P1 | **拖拽上传** | 支持文件拖拽到上传区域 | Unstructured |
| P1 | **搜索增强** | 角色全文搜索、别名搜索、按章节筛选 | — |
| P2 | **角色关系图谱** | 力导向图展示角色关系（基于共现分析） | PlotNovel |
| P2 | **统计仪表盘** | 角色数量趋势、置信度分布、审核完成度 | Label Studio |
| P2 | **暗黑模式** | 主题切换 | — |

### 3.2 架构与体验优化

| 优先级 | 优化项 | 说明 |
|--------|--------|------|
| P1 | **引入路由** | 使用 React Router 管理页面，支持 URL 直接定位到某本书的审核页 |
| P1 | **状态管理** | 引入 Zustand / Jotai 管理全局状态，减少 prop drilling |
| P1 | **组件库** | 引入 Headless UI / Radix UI + Tailwind，统一交互组件（Modal、Toast、Dropdown） |
| P2 | **虚拟列表** | 角色/书籍数量大时使用虚拟滚动优化性能 |
| P2 | **键盘快捷键** | 审核页支持 `A`(Approve) / `R`(Reject) / `→`(下一个) 等快捷键 |
| P2 | **响应式优化** | 移动端适配审核卡片布局 |

### 3.4 UI 设计参考建议

| 元素 | 建议 |
|------|------|
| **配色** | 当前灰白系可延续，增加语义色（Pending-黄、Approved-绿、Rejected-红、Extracting-蓝） |

## 五、总结

当前 novel-agent 的前端功能较为基础，主要完成了"能用"的目标。对比同类文档解析和文本标注平台，差距主要体现在：

1. **提取流程的可视化不足**：用户无法直观看到管道各阶段的执行状态
2. **审核效率工具缺失**：缺少原文对照、来源定位、批量操作等提升审核效率的关键功能
3. **结果展示维度单一**：仅有角色列表，缺少统计洞察和关系可视化
4. **交互细节粗糙**：无拖拽、无快捷键、无路由

建议优先实施 **第一阶段** 和 **第二阶段** 的功能，这将显著提升用户上传-提取-审核的核心链路体验。第三阶段的图谱和仪表盘可作为差异化亮点后续迭代。
